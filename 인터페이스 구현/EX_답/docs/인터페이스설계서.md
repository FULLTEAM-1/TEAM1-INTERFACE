## 1. 인터페이스 개요

### 1.1 목적
인사관리시스템(HRM)에서 **신규 사원 입사 등록**이 완료되는 시점에,
그룹웨어 시스템(Groupware)으로 사원 정보를 전달하여
**전사 계정(메일/메신저/결재용 계정)을 자동 발급**한다.

### 1.2 인터페이스 방식
**HRM 시스템과 Groupware 시스템은 서로 다른 부서에서 운영**되므로 직접 HTTP 호출이 어렵다.
중간에 **인터페이스 테이블(if_outbox / if_inbox)** 을 두어 **비동기 배치** 로 처리한다.

### 1.3 시스템 구성도

#### 1.3.1 시스템 구성 요약

| 구분 | 송신 시스템 (HRM) | 인터페이스 영역 | 수신 시스템 (Groupware) |
|:---:|:---|:---:|:---|
| **역할** | 입사 정보 적재 | 비동기 큐 (DB 테이블) | 계정 자동 발급 |
| **주요 구성** | 입사등록 폼(JSP)<br>EmpRegistServlet<br>EmployeeDAO | `if_outbox` (HrmDB)<br>`if_inbox` (GroupwareDB) | AccountBatchServlet<br>AccountDAO / InboxDAO |
| **데이터** | employee 테이블 | payload (JSON)<br>status (N / S / F) | account 테이블<br>account_history |
| **동작** | INSERT outbox | Polling (5분 주기 배치) | INSERT inbox -> account |

#### 1.3.2 데이터 흐름도

```
   [송신 시스템 HRM]                [인터페이스 테이블]              [수신 시스템 Groupware]
  +-------------------+            +-------------------+           +----------------------+
  | 입사등록 폼 (JSP) |            |                   |           | AccountBatchServlet  |
  |        |          |            |   if_outbox       |           |         |            |
  |        v          |  (1)INSERT |   (HrmDB)         |  (2)SELECT|         v            |
  | EmpRegistServlet  |----------->|  - outbox_id (PK) |---------->|   InboxDAO           |
  |        |          |            |  - tx_no          |  Polling  |         |            |
  |        v          |            |  - payload (JSON) |  (5분)    |         v            |
  | EmployeeDAO       |            |  - status (N/S/F) |           |   AccountDAO         |
  |        |          |            +-------------------+           |         |            |
  |        v          |                     ^                      |         v            |
  | employee 테이블   |            +-------------------+           |  account 테이블      |
  +-------------------+            |   if_inbox        |           |  account_history     |
                                   |   (GroupwareDB)   |  (3)INSERT|                      |
                                   |  - tx_no (UNIQUE) |<----------|                      |
                                   +-------------------+           +----------------------+
                                                                              |
                                            (4) UPDATE outbox.status = 'S/F'  |
                                   <------------------------------------------+
                       [모니터링 화면 : outbox / inbox / account 현황 + 실패 재시도]
```

#### 1.3.3 처리 단계 요약

| 단계 | 위치 | 동작 | 결과 |
|:---:|:---|:---|:---|
| (1) | HRM | 입사등록 -> `if_outbox` INSERT (status='N') | 미처리 건 적재 |
| (2) | Groupware | 5분 주기 배치 -> `status='N'` 건 SELECT | 처리 대상 추출 |
| (3) | Groupware | `if_inbox` + `account` + `account_history` INSERT | 계정 발급 완료 |
| (4) | HRM | `if_outbox.status` -> 'S'(성공) / 'F'(실패) UPDATE | 송신측 상태 동기화 |

## 3. 인터페이스 테이블 설계

### 3.1 송신 outbox (`HrmDB.if_outbox`)

| 컬럼 | 타입 | 설명 |
|:---|:---|:---|
| outbox_id | BIGINT AI PK | 송신 식별자 |
| if_id | VARCHAR(20) | IF_HR_001 |
| tx_no | VARCHAR(20) | 멱등키 = `emp_id` |
| payload | TEXT | 본문 JSON (아래 §3.3) |
| status | CHAR(1) | N: 미처리, S: 성공, F: 실패 |
| retry_cnt | INT | 재시도 횟수 (max 3) |
| reg_dt | DATETIME | 적재 일시 |
| proc_dt | DATETIME | 처리 완료 일시 |
| err_msg | VARCHAR(500) | 실패 사유 |

### 3.2 수신 inbox (`GroupwareDB.if_inbox`)

| 컬럼 | 타입 | 설명 |
|:---|:---|:---|
| inbox_id | BIGINT AI PK | 수신 식별자 |
| if_id | VARCHAR(20) | IF_HR_001 |
| tx_no | VARCHAR(20) **UNIQUE** | 멱등키 (중복 처리 방지) |
| payload | TEXT | 원본 본문 |
| status | CHAR(1) | S: 성공, F: 실패 |
| proc_ms | INT | 처리시간(ms) |
| recv_dt | DATETIME | 수신 일시 |
| err_msg | VARCHAR(500) | 실패 사유 |

> **[중요]** `if_inbox.tx_no` UNIQUE 제약 -> 같은 사원 정보가 두 번 들어와도 두 번째는 SQLException(중복키) 으로 차단되어 계정이 두 번 생성되지 않음 (**멱등성 보장**).

### 3.3 payload JSON 레이아웃

```json
{
  "empId": "E20260518001",
  "empName": "김신입",
  "deptCd": "D001",
  "position": "사원",
  "hireDt": "20260518",
  "email": "kim.newbie@company.com"
}
```

| 항목 | 영문명 | Type | 길이 | 필수 | 비고 |
|:---|:---|:---:|:---:|:---:|:---|
| 사원번호 | `empId` | String | 12 | Y | 멱등키 (E + YYYYMMDD + 일련) |
| 사원명 | `empName` | String | 50 | Y | UTF-8 |
| 부서코드 | `deptCd` | String | 4 | Y | D001~D999 |
| 직급 | `position` | String | 20 | Y | 사원/대리/과장/차장/부장 |
| 입사일 | `hireDt` | String | 8 | Y | YYYYMMDD |
| 이메일 | `email` | String | 100 | N | 계정 발급용 |

## 4. 처리 흐름

### 4.1 송신측 (HRM 입사 등록 시)

```
1. 입사등록 폼 → POST /send/emp
2. empId 자동 생성: "E" + YYYYMMDD + 3자리 일련번호
3. HrmDB 트랜잭션 시작
   3-1. INSERT employee
   3-2. INSERT if_outbox (status='N', payload=JSON, tx_no=empId)
   3-3. COMMIT
4. 사용자 화면: "입사 등록 완료 — 잠시 후 계정 발급됩니다"
```

### 4.2 수신측 (Groupware 배치)

```
1. 5분마다(또는 수동) AccountBatchServlet 실행
2. SELECT * FROM if_outbox WHERE status='N' AND retry_cnt < 3 LIMIT 100
3. 각 건마다:
   3-1. payload JSON 파싱
   3-2. GroupwareDB 트랜잭션 시작
        a. INSERT if_inbox (tx_no = empId) ← UNIQUE 제약으로 중복 차단
        b. INSERT account (
             account_id = empId,
             pwd_hash   = SHA-256(empId + "1234"),  ← 초기비번
             dept_cd    = deptCd,
             status     = 'A'  (활성)
           )
        c. INSERT account_history (생성 이력)
        d. COMMIT
   3-3. HrmDB: UPDATE if_outbox SET status='S' WHERE outbox_id=?
   3-4. 실패 시: UPDATE if_outbox SET status='F', retry_cnt+1, err_msg=?
```

## 5. 응답코드 정의

| 코드 | 의미 | 재시도 |
|:---:|:---|:---:|
| `00`  | 정상 | - |
| `E10` | 필수값 누락 | X |
| `E11` | JSON 파싱 오류 | X |
| `E20` | 멱등키 중복 (이미 발급된 계정) | X — outbox=S 로 마감 |
| `E30` | 부서코드 미존재 | 보류 — 부서 마스터 동기화 후 가능 |
| `E31` | 직급 미지원 (사원/대리/과장/차장/부장 외) | X |
| `E32` | empId 형식 오류 (E + 8자리 + 3자리) | X |
| `E90` | DB 오류 | O 자동 재시도 |
| `E99` | 알 수 없는 오류 | O |

## 6. 보안 (학습모듈 학습 2-3)

| 항목 | 적용 |
|:---|:---|
| SQL Injection | 모든 SQL **PreparedStatement + ?** 바인딩 |
| 트랜잭션 | 송신측: employee + if_outbox / 수신측: inbox + account + history |
| 멱등성 | `if_inbox.tx_no` UNIQUE |
| 비밀번호 | 초기 비번을 **SHA-256 해시** 후 저장 (평문 저장 금지) |
| 외부 노출 방지 | account.pwd_hash 는 모니터링 화면에 표시 X |
| 로그 보관 | outbox / inbox 양쪽에 payload 보관 (감사용) |

## 7. 검증 시나리오 (TC01~TC08)

| TC# | 케이스 | 입력 | 기대 결과 |
|:---:|:---|:---|:---|
| TC01 | 정상 입사 등록 | empName=김신입, deptCd=D001, position=사원 | outbox=S, inbox=S, account 생성, pwd_hash 저장됨 |
| TC02 | 같은 empId 재발급 시도 | 같은 empId 로 outbox 한 건 더 INSERT (강제) | inbox UNIQUE 위반 -> outbox=S(SKIP), account 변화 없음 |
| TC03 | 부서코드 없음 | deptCd=D999 (미등록) | outbox=F (E30), retry_cnt=1 |
| TC04 | 직급 미지원 | position=인턴 | outbox=F (E31) |
| TC05 | 필수값 누락 | empName 빈값 | 송신측 거부 (E10) |
| TC06 | 수신측 다운 후 복구 | 배치 안 돌리고 5건 등록 -> 1시간 후 배치 | outbox 누적 -> 일괄 처리 (5건 모두 S) |
| TC07 | retry 3회 초과 | TC03 케이스 그대로 3회 재시도 | retry_cnt=3, 더 이상 잡히지 않음 -> 모니터링 알림 |
| TC08 | 수동 재처리 | TC03 의 F건 -> 부서코드 D999 DB 추가 후 [재시도] | outbox N->S 전환 |

## [조장] 설계 / 공통 / 통합 — 20점

1) [ ] 인터페이스 설계서(IF_HR_001)에서 송신/수신 **공통 데이터 6항목** (emp_id, name, dept_cd, position, hire_dt, email) 매핑표를 작성했는가 **(5점)**

2) [ ] DB 스키마: HrmDB / GroupwareDB 2개 분리 + **`if_outbox.tx_no UNIQUE`** + **`account.dept_cd FK->dept`** 제약이 적용되었는가 **(5점)**

3) [ ] 공통 모듈(`DBManager.java`, `HashUtil.java`) 가 HRM/GW 각각 정상 동작하고 `db.properties` 가 환경에 맞게 설정되었는가 **(5점)**

4) [ ] 조원 결과 통합 후 Tomcat 배포 정상 + **검증 보고서** + before/after 캡처 8컷 포함 제출 zip 작성하였는가 **(5점)**

## [조원1] 송신측 구현 — 20점

5) [ ] `Employee` VO + `EmployeeDAO.insert()` + `nextEmpId()` 가 PreparedStatement 로 정상 동작하는가 **(5점)**

6) [ ] `OutboxDAO.insert()` 가 status='N' + payload JSON 으로 if_outbox 에 정상 적재되는가 **(5점)**

7) [ ] `EmpRegistServlet` 가 **employee + if_outbox 를 단일 트랜잭션**(setAutoCommit(false) -> commit/rollback)으로 처리하고 결과 화면에 empId/outboxId 전달하는가 **(10점)**

## [조원2] 수신측 배치 구현 — 20점

8) [ ] `AccountDAO.insert()` 가 pwd_hash 컬럼만 사용(평문 저장 X) + `InboxDAO.existsByTxNo()` 멱등 체크가 동작하는가 **(5점)**

9) [ ] `AccountBatchServlet` 가 outbox polling -> **SHA-256 해시 + dept FK 검증 + account/account_history/inbox 적재 + outbox 'S' 갱신** 을 단일 트랜잭션으로 처리하는가 **(10점)**

10) [ ] 멱등성: 동일 tx_no 재배치 시 **계정 중복 생성 안 됨** + 통계 `스킵` 카운터 정상 증가하는가 **(5점)**

## [조원3] 모니터링 / 예외처리 / 보안 — 20점

11) [ ] `/monitor/list` 가 outbox/inbox/account 3개 테이블 + 통계 + 처리시간(procMs) 실시간 표시 + status 색상 구분 + 실패건에만 재시도 버튼 노출하는가 **(10점)**

12) [ ] 실패 시나리오: dept_cd=D999 입력 -> outbox 'F' + err_msg 기록 -> 재시도 버튼 클릭 -> 'N' 복원 -> 부서 추가 후 재배치 'S' 전환이 검증되었는가 **(5점)**

13) [ ] 보안: 전 SQL **PreparedStatement + ? 바인딩** + 비밀번호 **SHA-256 해시** + 모니터링 화면에 pwd_hash 미노출이 모두 적용되었는가 **(5점)**
